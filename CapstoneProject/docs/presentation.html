Core Workflow

RAG Pipeline — End to End

📥 INGESTION

1Upload PDF / DOCX / CSV / TXT / JSON → stored on disk
2Parse — pdfplumber · python-docx · Elsevier JSON parser
3Chunk — 500-char windows, 50-char overlap (LangChain splitter)
4Embed + store — all-MiniLM-L6-v2 → ChromaDB with rich metadata

💬 QUERY

5Embed query with the same model
6Retrieve top-5 chunks (cosine similarity)
7Prompt = system prompt + last 3 exchanges + context + question
8Generate — Ollama/gemma returns grounded answer + citations

Fallback: "The uploaded documents do not contain sufficient information to answer this question."

Lessons Learned

Challenges & Solutions

🔴
ChromaDB HNSW index corruption
Running the seed script and backend concurrently wrote two segment directories, crashing the index. → Rule: always complete seeding before starting the backend. Added graceful HNSW guard in code.
🟡
CPU inference latency
gemma on CPU takes 30–90 s per query. Answer quality is high but speed is a demo constraint. → Set generous client timeouts (180 s); added spinner UX; Groq fallback identified as next step.
🟡
Chunk size vs. retrieval precision trade-off
Large chunks improve context but reduce retrieval precision; small chunks lose context. → Settled on 500 chars / 50 overlap; scientifically-tuned for abstract + body text.
🟢
Port & CORS management
Conflicting Uvicorn/Streamlit processes across sessions required careful port assignment. → Documented canonical ports (API: 8501, UI: 8001) in README and CORS config.
